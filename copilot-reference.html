GitHub Copilot Mastery — Reference

Everything covered so far, in one place. Bookmark this page for daily reference.

Module 1 — Setup & First Suggestion Done

See Copilot turn a plain-English comment into working code.

Key concept

Copilot predicts code based on: your comment's exact wording, the filename/language, and patterns from its training data. It is not reading your mind — always review what it generates.

Shortcuts

ActionShortcut
Accept a suggestionTab
Reject/dismiss a suggestionEsc
Next alternativeAlt+] / Option+]
Previous alternativeAlt+[ / Option+[
Force a suggestionAlt+\ / Option+\

Run a Python file

python3 filename.py
Lesson learned: Copilot sometimes adds extra unrequested code (it pattern-matches to common examples). Always read and clean up what it gives you before trusting it.

Module 2 — Prompting Quality Done

Understand how comment specificity changes Copilot's output quality.

The three levels

  1. Vague comment (# sort the list) → Copilot guesses, often wrong, silently.
  2. Specific comment (# sort a list of dictionaries by the 'price' key, descending) → correct logic, generic naming.
  3. Typed function signature (def sort_products_by_price(products: list[dict]) -> list[dict]:) → best result: clear names, correct logic, often auto-generates a docstring.

Vocabulary

Type hint: annotation telling Python what data type a variable/parameter should be, e.g. products: list[dict]. Doesn't change runtime behavior — it's documentation a machine can also check.

Lambda: a small, one-line, unnamed function. lambda x: x['price'] means "given x, return x['price']." Often used inline as a sort key.

Lesson learned: Specificity + structure = your biggest lever for good Copilot output.

Module 3 — Copilot Chat Done

Use conversational AI (not autocomplete) to explain and fix code.

Open Chat

Ctrl+Alt+I   (Win/Linux)
Cmd+Ctrl+I   (Mac)

Slash commands

CommandWhat it does
/explainExplains selected code in plain English, including hidden risks
/fixDiagnoses and proposes a fix for a bug
/testsGenerates test cases for a function
@workspaceBrings in context from the entire project

Vocabulary

Traceback: Python's error report showing the chain of calls leading to a crash — read bottom to top.

KeyError: a crash from accessing a dictionary key that doesn't exist.

.get('key', default): the safe way to read a dictionary — returns a default instead of crashing.

Clean file overwrite trick (fixes editor corruption)

cat > filename.py << 'EOF'
[paste your full clean code here]
EOF
Remember: "runs without crashing" ≠ "correct." Always check whether a fix reflects the right behavior, not just the simplest technical patch.

Module 4 — Custom Instructions Done

Set project-wide rules Copilot follows automatically.

Create the instructions file

mkdir -p .github
cat > .github/copilot-instructions.md << 'EOF'
- All Python functions must have type hints and docstrings
- When accessing dictionary keys that might be missing, always use .get() with a sensible default instead of square-bracket access
- Prefer list comprehensions over explicit for-loops where it keeps the code readable
EOF

Verify it's working

Ask Copilot for a new function with zero style guidance in your prompt — if it still follows your rules automatically, it's working.

Type hint vs. Docstring

Type hintDocstring
What data types go in/outWhat the function actually does
Machine-checkable, terseHuman sentences, more detail
Lesson learned: Custom instructions are a strong nudge, not a guarantee. Still review every output.

Module 5 — Agent Mode & Testing Done

Delegate multi-step, multi-file tasks and review autonomous AI work.

Key concept

Agent Mode = delegation, not assistance. Describe a goal; it plans, creates/edits multiple files, and may propose terminal commands — pausing for approval at each step (depending on settings).

Watch for: Agent may add behavior you didn't ask for (e.g. input validation). Not wrong — but a decision it made unilaterally. Always identify what it added beyond your literal request.

What a test is

Code that automatically checks whether other code behaves correctly, using assert statements — replacing manual "run it and eyeball the output" checking.

Anatomy of a test

def test_add_item_to_empty_inventory():   # name describes the scenario
    inventory = []                         # set up starting condition
    updated = add_item(inventory, "Widget", 9.99, 5)  # call the real function
    assert len(updated) == 1               # the actual claim being checked

Run tests

pytest

Reading pytest output

collected 5 items                                    ← total test_ functions found
test_inventory.py .....                    [100%]    ← one dot per PASSED test; F = failed
5 passed in 0.02s                                     ← final summary — always check this

Prove a test suite is real

  1. Deliberately introduce a small wrong change ("fake bug")
  2. Run pytest again — confirm it shows F
  3. Undo the fake bug, confirm all pass again

Safe find-and-replace (used to inject/remove a fake bug)

sed -i 's/OLD_TEXT/NEW_TEXT/' filename.py
Lesson learned: A passing test only proves the code satisfies that specific check — not that it does everything you intended.

Module 6 — Copilot CLI Done

Use Copilot directly from the terminal, outside any editor UI.

Command format

copilot -p "your plain-English task description"

-p = "prompt mode" — runs once, non-interactively, then exits.

When to use CLI vs. the editor

  • No graphical editor available (e.g. SSH'd into a server)
  • Quick one-off tasks while already in a terminal
  • Scripting/automating repetitive tasks
  • Managing multiple projects without opening each in a full editor
Safety behavior: in -p mode, CLI cannot ask permission mid-task (no one is present to respond) — so it refuses to run commands entirely rather than assume "yes." The more autonomous a mode is, the more cautious it should be.

Module 6.5 — Git & PR Workflow Done

Move from working directly on main to a safe branch → commit → push → PR flow.

Vocabulary

TermMeaning
BranchAn isolated parallel copy of your code — experiment safely without touching main
CommitA checkpoint snapshot of changes, saved locally, with a description
PushSends local commits up to GitHub, making them visible online
Pull Request (PR)A formal request to merge your branch into main, with room for review/checks
MergeActually combining your branch's changes into main

Create and switch to a new branch

git checkout -b branch-name

Check current status / branch

git status

Notice: your terminal prompt also shows the current branch in parentheses, e.g. (add-negative-price-check).

The core save cycle

git add .
git commit -m "short description of what changed"
git push
StepAnalogy
git add .Putting pages into an envelope, ready to mail
git commit -m "..."Sealing the envelope, writing a label
git pushActually mailing it — now it exists elsewhere too
Common typo traps: git add. (missing space) fails. git commit "msg" (missing -m) fails. orgin instead of origin fails.

First push of a brand-new branch

If you see "has no upstream branch", run the command Git suggests:

git push --set-upstream origin branch-name

Only needed the first time you push a new branch — after that, plain git push works.

.gitignore — exclude files Git should never track

echo "__pycache__/" > .gitignore

Reading git status output

Untracked filesGit sees them but isn't saving history yet
Changes to be committedStaged, ready for commit
ahead of 'origin/main' by N commitsLocal commits GitHub doesn't have yet — need to push
nothing to commit, working tree cleanEverything saved and in sync

Opening a PR

After pushing a new branch, GitHub gives you a direct link in the terminal output, e.g.:

https://github.com/USERNAME/REPO/pull/new/branch-name

Open that URL in your browser (not the terminal). Fill in title + description, review the diff (red = removed, green = added), then click Create pull request.

Reading a PR diff

  • base = branch you're merging INTO (usually main)
  • compare = your branch with the new changes
  • Red line = removed, Green line = added, white = unchanged context
  • "Able to merge / No conflicts" = safe to combine automatically
Real workflow order: Open PR → automated checks run (Module 7) → review happens (Module 8) → then merge. Don't merge immediately after opening.

Module 7 — CI/CD with GitHub Actions Done

Make tests run automatically on every Pull Request, instead of relying on remembering to run pytest by hand.

Key concept

CI = Continuous Integration — automatically verifying every proposed change (every PR) the moment it's opened, on a brand-new, disposable cloud machine. Proves the code works in a clean environment, not just "worked on my machine."

Nuance: the workflow makes tests run and show ✅/❌ — it does not by itself block the merge button. Actually enforcing "no merge until green" needs a separate branch protection rule.

Where the recipe lives

.github/workflows/ci.yml

The final, working workflow file

name: Run Tests

on: [pull_request]

jobs:
  test:
    runs-on: ubuntu-latest

    steps:
      - name: Check out code
        uses: actions/checkout@v4

      - name: Set up Python
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: Install pytest
        run: pip install pytest

      - name: Run tests
        run: pytest

Line-by-line meaning

LineMeaning
name: Run TestsLabel shown in GitHub's Checks tab / Actions dashboard
on: [pull_request]Trigger — run whenever a PR is opened or updated
runs-on: ubuntu-latestUse a fresh, disposable Ubuntu Linux machine
uses: actions/checkout@v4Official pre-built action: download the repo's code onto that machine
uses: actions/setup-python@v5Official pre-built action: install Python (version via with:)
run: pip install pytestDirect terminal command — installs pytest
run: pytestThe actual test run — same command you type manually

YAML indentation rule

Works like Python's indentation — consistent nesting defines structure. A step must be indented under steps:, which is indented under the job name, which is indented under jobs:. Get it wrong and the whole file fails to parse.

Safe way to write/fix a YAML file

cat > .github/workflows/ci.yml << 'EOF'
[paste the full correct YAML here]
EOF

Committing and pushing it

git add .
git commit -m "Add CI workflow to run pytest automatically on pull requests"
git push

Remember: stage → seal with a commit message → mail it with push. Skipping the commit means push has nothing new to send, even if staged.

Where to see it working

  • PR page → Checks tab → workflow name with a green ✅ or red ❌
  • Click in to see the full step-by-step log matching your YAML exactly
Lesson learned: real, automated tests ran on a machine you never touched, triggered purely by git push — the complete "workflow → PR → CI/CD" pipeline.

Module 8 — Code Review & Merge Done

Get an automated review on your PR, then merge it properly into main.

Request Copilot as a reviewer

PR page → right sidebar → "Reviewers" → click "Request" next to Copilot. It reads your actual diff/files (not just a summary) and leaves inline comments plus an overview.

Real issue caught in this project: scope creep — a PR titled "Improved error message" had grown to include a CI workflow and two docs files. Copilot flagged: "This scope mismatch makes it harder to review... consider updating the PR title/description or splitting into separate PR(s)."

Merged vs. Closed — critical distinction

StatusMeaning
OpenStill active, unresolved
Closed (no merge icon)Withdrawn — nothing went into main
Merged (purple badge)Successfully combined into main

Always verify a merge actually happened

git checkout main
git pull
cat filename.py   # confirm your change is really there

Module 9 — Cloud Agent Done

Assign a GitHub Issue and have Copilot independently create a branch, write code + tests, and open a PR.

How it differs from Agent Mode

Agent Mode (M5)Cloud Agent (M9)
Branch creationYou, manuallyCopilot, automatically
Code writingCopilot, with live approval per stepCopilot, working independently
PR openingYou, manuallyCopilot, automatically

How to trigger it

  1. Write a clear GitHub Issue: title + detailed description (signature, edge cases, reminder to follow .github/copilot-instructions.md)
  2. Assign the Issue to Copilot (right sidebar → Assignees)
  3. Copilot creates a branch, writes code + tests, opens a linked PR (often starts as "[WIP]"/"Draft")

Two safety gates before merging an agent PR

"1 workflow awaiting approval" — GitHub requires a human to explicitly approve running CI on agent-created PRs (protects against a compromised/malicious automated PR silently running code). Click "Approve workflows to run."
"Draft pull requests cannot be merged" — click "Ready for review" first.

Review discipline (same as always, matters more here)

  • Read the actual diff, not just Copilot's own summary
  • Check against the original issue requirements, item by item
  • Check against custom instructions (type hints, docstrings, .get())
  • Check out the branch and run tests yourself — don't just trust green checks or an AI's success claim
git fetch origin
git checkout name-of-agents-branch
pytest
Lesson learned: more autonomy on the AI's side means review discipline matters more, not less.

Complete Git Command Reference

Checking where you are

git status                  # current branch, staged/unstaged/untracked files
git branch                  # list all local branches (★ marks current one)
git branch -a               # list local AND remote branches
git log                     # full commit history (press 'q' to exit)
git log --oneline           # compact one-line-per-commit history

Creating and switching branches

git checkout -b new-branch-name     # create a NEW branch and switch to it
git checkout existing-branch-name   # switch to an EXISTING branch (no -b)
git switch new-branch-name          # newer alternative to checkout for switching
git switch -c new-branch-name       # newer alternative to checkout -b
Key distinction: -b = "build/create new." Leaving it off = "just switch to something that already exists." Using -b on an existing branch gives: fatal: a branch named 'X' already exists.

The core save cycle (do this constantly)

git add .                        # stage everything changed
git add filename.py              # stage just one specific file
git commit -m "clear description of what changed"
git push                         # send commits to GitHub (existing branch)
git push --set-upstream origin branch-name   # first push of a BRAND NEW branch only

Getting the latest changes from GitHub

git fetch origin        # download info about new branches/commits, WITHOUT merging
git pull                # download AND merge the latest changes into your current branch

Difference: fetch just looks and reports what's new remotely (safe, doesn't touch your files). pull = fetch + automatically merges into what you're currently on.

Checking out someone else's branch (e.g. reviewing a Cloud Agent's PR)

git fetch origin
git checkout branch-name-from-github

Undoing / cleaning up (use carefully)

git restore --staged filename.py    # un-stage a file (after add, before commit)
git restore filename.py             # discard uncommitted changes (careful — permanent)
git branch -d branch-name           # delete a local branch (only if already merged)
git branch -D branch-name           # force-delete a local branch (careful)

Comparing changes

git diff                    # unstaged changes (edited, not yet added)
git diff --staged           # staged changes (about to be committed)

Ignoring files permanently

echo "__pycache__/" > .gitignore     # create/overwrite with one pattern
echo "*.log" >> .gitignore           # APPEND another pattern (>> not >, or you'll erase the file)

Common typo traps (all seen firsthand this session)

TypoProblem
git add.Missing space — not a real command
git commit "message"Missing -m flag
git push --set-upstream orgin branchMisspelled "origin"
git checkout -b existing-branch-b fails if branch exists — drop it to just switch

Full real-world sequence, start to finish

git checkout -b my-feature-branch          # 1. create a branch
# ... make your code changes ...
git add .                                  # 2. stage everything
git commit -m "describe the change"        # 3. commit locally
git push --set-upstream origin my-feature-branch   # 4. push (first time only)
# ... open a PR on GitHub, get it reviewed/merged ...
git checkout main                          # 5. switch back to main
git pull                                   # 6. pull the now-merged changes
